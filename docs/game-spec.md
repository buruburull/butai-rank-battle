> **バージョン**: 1.0
> **最終更新**: 2026-03-11
> **プラットフォーム**: Minecraft Java Edition (Paper 1.21+)
> **ジャンル**: 競技型PvPプラグイン
> **開発**: BUTAI

---

## 1. ゲームコンセプト

### 1.1 概要

BUTAI RANK BATTLE（BRB）は、「フレーム」と呼ばれる特殊兵装を駆使して戦う競技型PvPゲームです。プレイヤーは戦闘スタイルに応じたフレーム（兵装モジュール）を装備し、エーテル（戦闘エネルギー）を管理しながら、ランクマッチで対戦します。シーズン制のランキングシステムにより、継続的な競技環境を提供します。

### 1.2 世界観

近未来、人類は「エーテル」と呼ばれるエネルギー物質を発見した。エーテルを動力源とする戦闘兵装「フレーム」が開発され、これを用いた競技「ランクバトル」が世界的な競技として確立された。プレイヤーは「オペレーター」として参戦し、ランクを上げていく。

### 1.3 用語定義（名称対応表）

| 旧名称（WT由来） | 新名称 | 説明 |
|---|---|---|
| トリガー | **フレーム (Frame)** | 装備する特殊兵装の総称 |
| トリオン | **エーテル (Ether)** | 戦闘用エネルギー。全兵装の動力源 |
| ベイルアウト | **エマージェンシーシフト (E-Shift)** | エーテル枯渇時の緊急離脱機構 |
| ロードアウト | **フレームセット (FrameSet)** | フレーム装備構成（8スロット） |
| Kogetsu等の武器名 | オリジナル名 | 下記フレーム一覧参照 |

※以下の用語はそのまま維持: RP、ランク、シーズン、チーム

## 2. プレイヤーシステム

### 2.1 オペレーター登録

サーバー初参加時に自動登録されます。プレイヤーは「オペレーター」としてシステムに記録されます。

初期ステータス：
- ランク: UNRANKED（未所属）
- RP: 各武器タイプ1000（初期値）
- エーテル上限: 1000（成長システムにより最大2000まで拡張可能）
- フレームセット: 空（未装備）

### 2.2 ランク（階級）システム

総合RP（全武器RPの合計）に基づいてランクが決定されます。

| ランク | 必要総合RP | 表示色 | 表示名 |
|---|---|---|---|
| S | 15,000+ | 金（太字） | S級 |
| A | 10,000+ | 赤 | A級 |
| B | 5,000+ | 水色 | B級 |
| C | 5,000未満 | 緑 | C級 |
| UNRANKED | 初期状態 | 白 | 未所属 |

ランクは試合ごとに自動再計算されます。

### 2.3 RP（ランクポイント）システム

RPは武器タイプ別に独立して追跡されます。

武器タイプ：
- **STRIKER（ストライカー）**: 近接戦闘系フレーム
- **GUNNER（ガナー）**: 射撃系フレーム
- **MARKSMAN（マークスマン）**: 狙撃系フレーム

#### RP計算式（非対称Elo方式）

勝者のRP獲得:
```
base = 30
coefficient = 1.0 + (敗者RP - 勝者RP) / 1000
winnerGain = clamp(base * coefficient, 5, 120)
```

敗者のRP喪失:
```
loserLoss = round(winnerGain * 0.7)  // 勝者の70%のみ失う
loserLoss = max(5, loserLoss)
```

参加ボーナス:
```
全参加者に +5 RP
```

計算例（同RP同士の場合）:
- 勝者: +35 RP（30 + 5参加ボーナス）
- 敗者: -16 RP（-21 + 5参加ボーナス）
- 1マッチあたりのシステム注入: +19 RP

この非対称設計により、プレイヤー全体のRP総量が試合ごとに増加し、継続プレイでランク昇格が可能になります。

### 3.2 フレームセット（装備構成）

プレイヤーは8つのスロット（メイン4枠 + サブ4枠）にフレームを装備できます。

#### スロット構成

| スロット | 区分 | 用途 |
|---|---|---|
| 1（メイン武器） | メイン | **必須**。武器系フレーム（STRIKER/GUNNER/MARKSMAN）のみ。RPに影響する武器タイプを決定 |
| 2〜4 | メイン | 武器系 or SUPPORTフレームを自由に配置 |
| 5〜8 | サブ | 武器系 or SUPPORTフレームを自由に配置 |

#### 装備制約ルール

1. **重複禁止**: 同一フレームは1つまで（例: Leap×2は不可）
2. **スロット1必須**: スロット1には武器系フレーム（STRIKER/GUNNER/MARKSMAN）を必ず装備すること。空の状態ではマッチにキューできない
3. **武器タイプ決定**: スロット1に装備したフレームのカテゴリが、そのマッチでのRP変動対象の武器タイプとなる
   - 例: スロット1にCreescent（STRIKER）→ STRIKER RPが変動
   - 例: スロット1にFalcon（MARKSMAN）→ MARKSMAN RPが変動
4. **メイン/サブの区別**: メイン枠（1〜4）とサブ枠（5〜8）にカテゴリ制限はないが、メイン枠のフレームはホットバーの左側（アクセスしやすい位置）に配置される
5. **最低装備数**: スロット1の武器系フレーム1つが最低条件。スロット2〜8は空でも可

#### バリデーション

マッチキュー参加時に以下をチェック:
- スロット1が武器系フレームで埋まっているか
- 同一フレームが複数装備されていないか
- 不正なスロットにフレームが配置されていないか

違反時はエラーメッセージを表示し、キュー参加を拒否します。

#### 操作コマンド

- `/frame set <スロット> <フレーム名>` - フレーム装備
- `/frame view` - 現在のフレームセット確認
- `/frame remove <スロット>` - フレーム解除（スロット1は武器系で上書きのみ可）
- `/frame list` - 全フレーム一覧
- `/frame preset save <名前>` - プリセット保存
- `/frame preset load <名前>` - プリセット読み込み
- `/frame preset list` - プリセット一覧

フレームセットはDB永続化され、セッション間で保持されます。

### 3.3 フレーム特殊効果（実装目標）

#### 近接系
- **Crescent**: 近接攻撃時にダメージ1.3倍補正
- **Fang**: 背面攻撃時に1.5倍補正（基本背面1.5倍と乗算で2.25倍）
- **Bastion**: 右クリックでシールドモード切替（攻撃力半減・被ダメ60%カット）

#### 射撃系
- **Pulse**: 標準弓射撃（エフェクトなし）
- **Nova**: 分散弾。3発の弾が散らばり、各着弾時に小爆発（半径2.5、威力3.0）
- **Seeker**: 投擲後、最寄りの敵に向かって軌道修正（射程20ブロック以内）
- **Frost**: 命中した敵にSlowness II（移動速度低下）を3秒間付与。ダメージは低いがSTRIKERとの連携に有効

#### 狙撃系
- **Falcon**: 2.0秒チャージで威力2.5倍。未チャージは通常威力
- **Volt**: 矢が最大3体の敵を貫通。1.2倍ダメージ補正
- **Zenith**: 3.0秒フルチャージで威力3.0倍。最大火力フレーム

#### 補助系
- **Leap**: 向いている方向にダッシュ。空中使用可
- **Cloak**: 持続トグル。有効中は被ダメ60%カット、毎秒15エーテル消費
- **Warp**: エンダーパールのように瞬間移動（射程32ブロック）
- **Vant**: 前方に3×3の壁を生成。4秒後に消滅
- **Blast**: TNTを設置・爆発。半径5.0、威力6.0
- **Tracer**: 命中した敵に発光（Glowing）効果5秒。チーム全員から視認可能

## 4. エーテル（エネルギー）システム

### 4.1 基本仕様

エーテルは試合中の全アクションの動力源です。

- **最大値**: 1000
- **表示**: 経験値バーで視覚化（レベル数値 = エーテル残量、バー = 割合）
- **回復**: なし（試合中は回復しない）

### 4.2 エーテル消費

消費タイプは2種類あります。

**使用時消費（Use）**: フレーム発動ごとに一定量消費
- 例: Leap = 40/回、Warp = 60/回

**持続消費（Sustain）**: 有効中に毎秒消費し続ける
- 例: Cloak = 15/秒

### 4.3 エーテルリーク（HP連動消費）

ダメージを受けてHPが減少している間、エーテルが自動的に漏出します。

```
リーク量/秒 = (最大HP - 現在HP) * 0.5
```

この仕組みにより、HPの低いプレイヤーはエーテルも急速に失い、早期決着を促します。

### 4.4 警告システム

| エーテル残量 | 演出 |
|---|---|
| 200以下 | 黄色警告メッセージ |
| 100以下 | 赤色点滅警告 |
| 0 | エマージェンシーシフト発動 |

### 4.5 エマージェンシーシフト（E-Shift）

エーテルが0になるとE-Shiftが自動発動します。

動作:
1. プレイヤーを即座にロビーへテレポート（キルではない）
2. 試合システムに脱落を通知
3. 二重発動防止フラグを設定

E-Shiftはプレイヤーの「死亡」ではなく「脱落」として処理されます。

## 5. マッチシステム

### 5.1 マッチの種類

#### ソロランクマッチ
- 参加: 1人ずつキューに参加、2人揃ったら開始
- 制限時間: 5分（300秒）
- 勝利条件: 相手を倒す or 制限時間終了時にジャッジスコアが高い方が勝利（後述）
- RP変動: あり（非対称Elo方式）

#### チームランクマッチ
- 参加: チームリーダーがチームごとキューに参加
- 制限時間: 10分（600秒）
- 勝利条件: 相手チームの全員を脱落させる or 制限時間終了時にチーム合算ジャッジスコアで判定（後述）
- RP変動: あり（チーム用計算式）

#### プラクティスバトル
- 参加: 1人ずつキューに参加
- 制限時間: 5分
- 勝利条件: 同上
- RP変動: なし（練習用）

### 5.2 制限時間切れ時の勝敗判定（ジャッジシステム）

制限時間内に決着がつかなかった場合、以下の2段階で勝敗を決定します。

#### 第1段階: ジャッジスコア判定

各プレイヤーの **ジャッジスコア** を算出し、高い方が勝利。

```
ジャッジスコア = 与ダメージ量 × 0.5 + エーテル残量 × 0.3 + HP残量 × 0.2
```

| 要素 | 重み | 理由 |
|---|---|---|
| 与ダメージ量 | 50% | アグレッシブなプレイを最も評価 |
| エーテル残量 | 30% | リソース管理の巧みさを評価 |
| HP残量 | 20% | 生存力を評価 |

チーム戦の場合: チーム内の生存メンバーのジャッジスコア合計で判定。脱落済みメンバーのスコアは0。残存人数が多い方が基本的に有利。

#### 第2段階: サドンデス（同点時）

ジャッジスコアが同点（差が1%以内）の場合、**サドンデス** に突入。

サドンデスのルール:
- エーテルリーク係数が **3倍** に加速（通常0.5 → 1.5/秒）
- 追加制限時間: **60秒**
- エーテル残量が少ない方から急速に脱落する
- 60秒以内に決着しない場合: **引き分け**（両者に参加ボーナス+5 RPのみ、勝敗RPなし）

サドンデス開始時の演出:
- ボスバーが赤色に変化
- 全プレイヤーに警告メッセージ: `§c§l⚠ サドンデス突入！エーテルリーク加速！`
- 効果音（ウィザースポーン音など）

### 5.3 マッチの流れ

```
キュー参加 → マッチング成立 → カウントダウン(10秒) → 試合開始 → 戦闘 → [制限時間切れ → ジャッジ判定 or サドンデス] → 試合終了 → 結果表示 → ロビーへ帰還
```

マッチ状態遷移:
```
WAITING → COUNTDOWN → ACTIVE → [JUDGE → SUDDEN_DEATH(任意)] → ENDING → FINISHED
```

### 5.4 マッチ中のルール

- 自然回復無効
- フレンドリーファイア無効（チーム戦）
- エーテルリーク有効
- 背面攻撃ボーナス有効
- ブロック変更の追跡と試合後復元（BlockChangeListener）

### 5.5 切断・ログアウト時の処理

試合中にプレイヤーが切断（ログアウト/タイムアウト）した場合の処理。

#### 即脱落

切断を検知した時点で **即座にE-Shift（脱落）扱い** とする。

- ソロ戦: 相手の勝利。通常の勝敗RP計算を適用
- チーム戦: 切断者は脱落扱い。チームの残りメンバーで試合続行

#### 切断ペナルティ（故意切断対策）

短期間に繰り返し切断した場合、キュー参加にペナルティを課す。

| 条件 | ペナルティ |
|---|---|
| 直近1時間以内に2回目の切断脱落 | 警告メッセージのみ |
| 直近1時間以内に3回目の切断脱落 | **5分間キュー参加禁止** |
| 直近1時間以内に4回以上の切断脱落 | **15分間キュー参加禁止** |

ペナルティ中にキューしようとした場合:
```
§c§l⚠ 切断ペナルティ中です。残り XX分 XXsでキュー参加可能になります。
```

※プラクティスバトルの切断はペナルティ対象外
※サーバー側の問題（サーバーリスタート等）による切断はペナルティ対象外

### 5.6 チーム戦のRP計算

```
勝利チーム: base(30) × coefficient + survivalBonus(生存:+10) + 貢献ボーナス(与ダメ20+:+5) + 参加ボーナス(5)
敗北チーム: -(base × 0.7) + survivalBonus(生存:+10) + 貢献ボーナス(与ダメ20+:+5) + 参加ボーナス(5)
coefficient = 1.0 + (敗北チーム平均RP - 勝利チーム平均RP) / 1000
```

※ソロ戦と同じ非対称Elo方式をベースに、個人貢献（生存・与ダメージ）を加味

### 5.7 マッチ結果表示

試合終了時に全プレイヤーに表示:
- マップ名
- 勝敗（ジャッジ判定の場合は「ジャッジ勝利」と表示）
- キル数
- ジャッジスコア（制限時間切れ時のみ）
- RP変動（緑:プラス / 赤:マイナス）
- ランキング（順位付き）

### 5.8 観戦（スペクテーター）システム

- `/rank spectate [matchId]` - 試合を観戦
- `/rank spectate leave` - 観戦終了
- 参加者と独立した観戦者セットで管理
- キュー中・試合中のプレイヤーは観戦不可

### 5.9 マッチ履歴

- `/rank history [ページ]` - 過去の戦績を閲覧
- 各マッチの詳細（マップ、参加者、キル、RP変動）をDB保存
- シーズンごとにフィルタ可能

## 6. ロビーシステム

### 6.1 ロビーの基本動作

- ログイン時: 必ずロビーにスポーン
- 試合終了時: ロビーにテレポート
- 死亡/リスポーン時: ロビーにテレポート
- ゲームモード: ADVENTURE（ブロック破壊不可）

### 6.2 NPC（案内ビレジャー）

ロビーにはインタラクション用NPCが配置されます。右クリックで対応アクションを実行します。

| NPC名 | アクション | 説明 |
|---|---|---|
| §6§lランクマッチ | ソロキュー参加 | `/rank solo` |
| §a§lプラクティス | 練習キュー参加 | `/rank practice` |
| §d§lチームマッチ | チームキュー参加 | `/rank team` |
| §b§lフレームセット | 装備確認 | `/frame view` |
| §e§l戦績 | 自分の戦績表示 | `/rank stats` |

NPC設定:
- AI無効、無敵、サイレント
- PersistentDataContainerでタグ管理（NPC_KEY にコマンド文字列を格納）
- Villager自体のCustomNameはnull（重複表示防止）
- 名前+サブタイトルはTextDisplayエンティティで頭上に表示（Billboard.CENTER）
- 座標・向きはframes.ymlで設定可能

### 6.3 ホログラム

TextDisplayエンティティでリアルタイム情報を表示します。

| ホログラム | タイプ | 更新間隔 |
|---|---|---|
| ウェルカムバナー | 静的テキスト | - |
| ランキングTOP10 | 動的（DB参照） | 60秒 |

ランキングホログラムは総合RPの上位10名を色分け表示します（1位:金、2位:白、3位:灰）。

### 6.4 アクションバー

## 8. シーズンシステム

### 8.1 概要

シーズンは一定期間のランキングを区切る仕組みです。シーズン終了時にスナップショットが保存され、全プレイヤーのRPがリセットされます。

### 8.2 シーズンの流れ

```
シーズン開始 → 期間中のランクバトル → シーズン終了 → RPスナップショット保存 → 全RPリセット(1000) → 次シーズン開始
```

### 8.3 管理コマンド

| コマンド | 説明 |
|---|---|
| `/bradmin season start <名前>` | シーズン開始 |
| `/bradmin season end` | シーズン終了（スナップショット+リセット） |

### 8.4 シーズン報酬（将来実装）

シーズン終了時のランクに応じた報酬を検討:
- S級: 限定称号、限定フレームスキン
- A級: 限定称号
- B級: 装飾アイテム
- 参加賞: シーズン参加バッジ

### 8.5 エーテル成長システム

プレイヤーは鉱山やMOBタワーでの活動を通じてエーテル上限を成長させることができます。

#### 成長パラメータ

| 項目 | 値 |
|---|---|
| 基本エーテル上限 | 1000 |
| 最大エーテル上限 | 2000 |
| 最大レベル | 40 |
| 1レベルあたり上限増加 | +25 |
| 必要EP計算式 | `100 × 1.10^Lv` |
| シーズンリセット | EP・レベルはリセット、エーテル上限は引き継ぎ |

#### EP（エーテルポイント）獲得方法

**鉱山エリア:**

| 鉱石 | EP |
|---|---|
| 石炭鉱石 | 1 |
| 鉄鉱石 | 2 |
| 金鉱石 | 3 |
| エメラルド鉱石 | 4 |
| ダイヤモンド鉱石 | 5 |

- 鉱石は採掘後、丸石に変わり30秒後に再生
- アイテムドロップなし（EP獲得のみ）

**MOBタワーエリア（階層式）:**

タワーは5階層構成で、上層ほど強力なMOBが出現しEPが高い。

| 階層 | MOB | 同時数 | EP | 解放レベル |
|---|---|---|---|---|
| 1F ゾンビの間 | ゾンビ | 8体 | 2 | 初期開放 |
| 2F スケルトンの間 | スケルトン | 8体 | 3 | Lv.5 |
| 3F 蜘蛛とウィッチの間 | クモ, ウィッチ | 10体 | 4-5 | Lv.15 |
| 4F 襲撃者の間 | ヴィンディケーター, ピリジャー | 10体 | 6-7 | Lv.25 |
| 5F ウォーデンの間 | ウォーデン | 3体 | 15 | Lv.35 |

- 各フロアにNPC配置：「次の階へ」（レベル不足時は拒否）「ロビーへ戻る」
- MOBは5秒間隔で自動スポーン
- プレイヤーが50m以内にいる時のみスポーン
- MOBからのドロップ・経験値なし（EP獲得のみ）

#### タワー/鉱山の共通ルール

- **インベントリ保存/復元**: エリア入場時にインベントリを保存し、退出時に復元
- **フレーム持ち込み禁止**: 入場時にインベントリをクリアし、鉄の剣（タワー）/ 鉄のピッケル（鉱山）を付与
- **デス時アイテムドロップ無効**: タワー/鉱山内でのデス時はアイテムをドロップしない
- **MOBドロップ無効**: タワー内のMOBはアイテム・経験値をドロップしない
- **鉱山はSURVIVALモード**: 鉱石のみ破壊可能（周囲のブロックは保護）

#### アクセス方法

ロビーの専用NPCを右クリックすると、各エリアにテレポート:
- **鉱山NPC** → 鉱山エリア（SURVIVALモード、鉄のピッケル付与）
- **MOBタワーNPC** → MOBタワー1Fエリア（鉄の剣付与）

## 14. 技術仕様

### 14.1 技術スタック

- サーバー: Paper 1.21+（Spigot fork）
- 言語: Java 21
- ビルド: Gradle 8.5 + Shadow Plugin
- DB: MySQL 8.0 / MariaDB
- 接続プール: HikariCP

### 14.2 プロジェクト構造

```
butai-rank-battle/
├── common/              # 共有ライブラリ（モデル, DAO, ユーティリティ）
├── core-plugin/         # メインプラグイン
│   ├── command/         # コマンドハンドラー
│   ├── listener/        # イベントリスナー
│   ├── manager/         # 各種マネージャー
│   └── arena/           # アリーナ管理
├── config/              # YAML設定
└── docs/                # ドキュメント・スキーマ
```

### 14.3 設計原則

- エーテル操作は必ずEtherManager経由（直接減算禁止）
- DB操作はPreparedStatement経由（SQLインジェクション防止）
- イベント処理は適切なpriorityとキャンセルチェック
- プレイヤーデータは必ずnullチェック
- プレイヤー向けメッセージは日本語、ログ/コードは英語
